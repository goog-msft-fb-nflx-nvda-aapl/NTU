01 · Overview

This report documents the implementation of a Uniform Grid acceleration structure in pbrt v4, along with a bonus Two-Level Grid extension. Both structures were integrated into pbrt v4's existing accelerator framework alongside BVH and KdTree, and evaluated across three production scenes.

02 · Uniform Grid Implementation

Files Modified

The implementation required changes to three files in the pbrt v4 source:

  • src/pbrt/cpu/aggregates.h — class declaration
  • src/pbrt/cpu/aggregates.cpp — constructor, Intersect, IntersectP, Create
  • src/pbrt/cpu/primitive.h — register in TaggedPointer type list

Grid Construction

The grid resolution is determined by the cube-root heuristic from pbrt v2: the voxel size is set so that the number of voxels along each axis is proportional to the scene's primitive count.

Float cubeRoot = std::cbrt((Float)prims.size());
Float voxelSize = MaxComponentValue(diag) / cubeRoot;
for (int i = 0; i < 3; ++i) {
    nVoxels[i] = std::max(1, (int)std::ceil(diag[i] / voxelSize));
    invWidth[i] = (diag[i] > 0) ? (nVoxels[i] / diag[i]) : 0;
}

Each primitive is inserted into all overlapping voxels using its bounding box. This may cause duplicates, but guarantees no intersection misses.

Ray Traversal (3D DDA)

Ray traversal uses the 3D Digital Differential Analyzer (DDA) algorithm — the same slab-stepping method used in voxel engines. For each axis, we precompute the parametric distances to the next voxel boundary (tNext) and the step size (tDelta). At each iteration, we advance along the axis with the smallest tNext.

int axis = (tNext[0] < tNext[1]) ? ((tNext[0] < tNext[2]) ? 0 : 2)
                                  : ((tNext[1] < tNext[2]) ? 1 : 2);
if (tNext[axis] > tMax) break;
voxel[axis] += step[axis];
if (voxel[axis] == out[axis]) break;
tNext[axis] += tDelta[axis];

This ensures each voxel is visited at most once per ray, and traversal stops as soon as a closer hit is found.

TaggedPointer Registration

pbrt v4 uses a tagged-pointer variant type for Primitive. Adding a new aggregate requires registering it in primitive.h:

class UniformGridAggregate;
// ...
: public TaggedPointer<SimplePrimitive, GeometricPrimitive, TransformedPrimitive,
                       AnimatedPrimitive, BVHAggregate, KdTreeAggregate,
                       UniformGridAggregate> {

And in CreateAccelerator():

else if (name == "uniformgrid")
    accel = UniformGridAggregate::Create(std::move(prims), parameters);

The accelerator can then be selected in any scene file with Accelerator "uniformgrid".

03 · Two-Level Grid (Bonus)

The basic uniform grid suffers from mailboxing — when primitives are large relative to voxel size, they are inserted into many voxels and tested repeatedly. The two-level grid addresses this by adding a second layer of spatial subdivision for dense cells.

Design

The top-level grid uses the same cube-root resolution heuristic. Each cell is classified at construction time:

  • Cells with ≤ 8 primitives store them directly (flat list)
  • Cells with > 8 primitives build a UniformGridAggregate sub-grid
struct TopCell {
    std::vector<Primitive> prims;            // sparse: direct list
    std::unique_ptr<UniformGridAggregate> subgrid; // dense: sub-grid
};

Ray Traversal

The outer traversal is identical to the uniform grid DDA. At each cell, rays are dispatched to either the sub-grid or the flat list:

if (cell.subgrid) {
    auto hit = cell.subgrid->Intersect(ray, tMax);
    if (hit) { si = hit; tMax = hit->tHit; }
} else {
    for (auto &prim : cell.prims) { ... }
}
Result: The two-level grid achieves rendering times close to BVH on all tested scenes, while maintaining construction times far below KdTree. On the Sportscar scene, it renders 7.5× faster than the flat uniform grid.

05 · Analysis

Construction Time

Uniform grid construction is dramatically faster than both BVH and KdTree. On Ganesha (4.3M prims), the grid constructs in 198 ms vs 16 seconds for KdTree — an 81× difference. This makes it attractive for scenes that change frequently (e.g., animation frames with deforming geometry).

Why is grid construction so fast? The algorithm is O(N) — each primitive touches a fixed average number of voxels. BVH uses recursive SAH splitting (O(N log N)) and KdTree is even more expensive due to sweep-based split finding.

Rendering Performance

The flat uniform grid is slower than BVH for rendering because it cannot adapt to non-uniform primitive distributions. When many triangles cluster in one region (e.g., Ganesha's intricate carvings), individual voxels become very dense, causing excessive intersection tests per ray.

Uniform grid weakness: Poor handling of non-uniform scenes. The Ganesha scene has an elongated grid (163×23×163) due to its flat base, concentrating primitives into a thin horizontal band of voxels.

Two-Level Grid Performance

The two-level grid resolves the density problem. On Sportscar, it is faster than BVH (0.96s vs 1.53s total). On Ganesha it achieves 2.7s vs BVH's 3.05s. The key insight is that sub-grids localize the primitive search within each dense region, dramatically reducing the average number of intersection tests per ray.

Two-level grid advantage: Combines the O(N) construction speed of uniform grids with much better ray traversal performance in non-uniform scenes. Memory usage is higher than a flat grid but lower than BVH.

Memory Usage

The uniform grid uses the least memory of all structures because it avoids tree node overhead. BVH requires ~2N nodes (each LinearBVHNode is 32 bytes), while the grid's voxel array has minimal per-cell overhead. The two-level grid uses slightly more memory than the flat grid due to sub-grid metadata.
