1. Source Code Change

The modification was made to a single function in src/pbrt/util/transform.h — the templated Normal3<T> Transform::operator() method (line 329). This is the function pbrt calls every time it transforms a surface normal through a Transform (e.g., when instancing geometry or applying object-to-world transforms to shading normals).

Mathematical Background

Points and vectors transform by the forward matrix M. Normals, however, must remain perpendicular to the surface after transformation. Under a non-uniform scale, applying M directly to a normal destroys this perpendicularity. The correct transform for normals is the inverse-transpose of M:

n' = (M-T) · n  =  transpose(M-1) · n

In pbrt's Transform class, the matrix and its inverse are stored as m and mInv. The inverse-transpose is obtained by using mInv but indexing it with swapped row/column indices — i.e., mInv[col][row] instead of mInv[row][col].

The Diff

The bug replaces the inverse-transpose computation with a direct forward-matrix transform — identical to how Vector3 is transformed: 

// File: src/pbrt/util/transform.h, line 329
template <typename T>
PBRT_CPU_GPU inline Normal3<T> Transform::operator()(Normal3<T> n) const {
    T x = n.x, y = n.y, z = n.z;
-   // CORRECT: uses mInv with transposed indices = inverse-transpose of M
-   return Normal3<T>(mInv[0][0] * x + mInv[1][0] * y + mInv[2][0] * z,
-                     mInv[0][1] * x + mInv[1][1] * y + mInv[2][1] * z,
-                     mInv[0][2] * x + mInv[1][2] * y + mInv[2][2] * z);
+   // BUG: uses m with normal row-major indices = same as Vector3 transform
+   return Normal3<T>(m[0][0] * x + m[0][1] * y + m[0][2] * z,
+                     m[1][0] * x + m[1][1] * y + m[1][2] * z,
+                     m[2][0] * x + m[2][1] * y + m[2][2] * z);
}
Key insight: The original code indexes mInv as mInv[col][row] — this is not a mistake, it is intentional transposition. The bug changes this to use the forward matrix m with standard m[row][col] indexing, making normals transform exactly like vectors.

2. Scene Design

The scene was designed to make the normal transformation bug as visible as possible. The key requirement is a non-uniform scale — uniform scaling and pure rotation do not break the buggy transform, because for those transforms M and (M-T) produce the same result up to a constant factor.

Design Choices

A unit sphere is scaled by 4× in the X direction only, producing a wide, flat ellipsoid. This is the worst case for normal errors: the normals on the narrow ends of the ellipsoid (near ±X) are nearly horizontal and are pulled sharply inward by the incorrect transform. A directional light from upper-right makes the shading errors dramatic and immediately visible.

    Shape "trianglemesh"
        "point3 P" [-10 0 -10  10 0 -10  10 0 10  -10 0 10]
        "integer indices" [0 1 2  0 2 3]
AttributeEnd
Why a 4× X-scale? For a scale matrix S = diag(4, 1, 1), the inverse-transpose is S-T = diag(1/4, 1, 1), while the forward matrix is diag(4, 1, 1). These differ maximally in the X direction, causing normals near the ends of the ellipsoid to point inward instead of outward — producing the dramatic concave artifact visible in the buggy render.

3 & 4. Renders: Buggy vs. Correct

Both images were rendered with identical scene files and settings (800×400, 128 samples/pixel, path integrator). The only difference is whether the normal transform uses the inverse-transpose or the forward matrix.

⚠ Buggy

Normals transformed like vectors (m[row][col]). The ellipsoid appears concave with a dark ring artifact — normals near the X-axis tips point inward, causing the surface to appear self-shadowed and inverted.

✓ Correct

Normals transformed by inverse-transpose (mInv[col][row]). The ellipsoid has smooth, physically correct shading that follows the surface curvature naturally.

Analysis of the Artifact

In the buggy render, the visible artifacts are:

  • Concave appearance: The center of the ellipsoid appears sunken inward, as if it were a bowl rather than a convex surface. This is because the normals in the center now point roughly in the +X direction (they were stretched by the 4× scale), rather than upward.
  • Dark ring artifact: A dark band appears around the perimeter of the ellipsoid where normals are nearly perpendicular to the light, creating a hard shadow terminator in the wrong location.
  • Bright outer rim: The thin outer edges of the ellipsoid appear bright orange — because the incorrectly transformed normals at the edges happen to face toward the light source even though the surface faces away.
Conclusion: The bug is immediately and unambiguously visible. The concave bowl illusion and the bright outer rim are impossible to achieve with correct normal transforms on a convex surface, making this an excellent demonstration of why normals require the inverse-transpose.
