01 — Overview

Task Description

The goal of this assignment is to build a Log Analysis Agent using the LangGraph framework. The agent receives raw system or authentication log lines as input and autonomously decides how to process each one based on its security severity.

The agent demonstrates agentic behavior through conditional branching: benign logs skip enrichment and go directly to report generation, while suspicious or critical logs are first enriched with threat intelligence before a report is produced.

All inference runs locally on an H200 server via Ollama with the llama3.2 model — no paid API keys required.

03 — Architecture

Agent Graph Design

The agent is implemented as a directed graph with three processing nodes and one conditional routing function. The graph is compiled using StateGraph from LangGraph.

INPUT
→
classify_log
⤵
enrich_threat
→
generate_report
→
END

Benign logs skip enrich_threat and branch directly to generate_report.

State Schema

The shared state object passed between all nodes is defined as a TypedDict:

class LogState(TypedDict):
    raw_log:        str   # input log line
    severity:       str   # "benign" | "suspicious" | "critical"
    threat_context: str   # enrichment from LLM (empty if benign)
    report:         str   # final formatted incident report
04 — Implementation

Node Descriptions

Node 1 — classify_log

The entry node. Passes the raw log to the LLM with a structured prompt and parses the single-word response (benign, suspicious, or critical). Falls back to suspicious if the model produces an unexpected response.

Node 2 — enrich_threat (conditional)

Only executed for suspicious or critical logs. Asks the LLM to provide a 2–3 sentence threat context covering: what attack is occurring, the attacker's goal, and recommended immediate action.

Node 3 — generate_report

The terminal node for all paths. Assembles the final structured incident report from the accumulated state fields. Benign logs receive an N/A threat context section.

Routing Function

def route_by_severity(state: LogState) -> Literal["enrich_threat", "generate_report"]:
    if state["severity"] in ("suspicious", "critical"):
        return "enrich_threat"
    return "generate_report"

Graph Construction

graph = StateGraph(LogState)

graph.add_node("classify_log",    classify_log)
graph.add_node("enrich_threat",   enrich_threat)
graph.add_node("generate_report", generate_report)

graph.set_entry_point("classify_log")

graph.add_conditional_edges(
    "classify_log",
    route_by_severity,
    {"enrich_threat": "enrich_threat", "generate_report": "generate_report"}
)
graph.add_edge("enrich_threat",   "generate_report")
graph.add_edge("generate_report", END)
05 — Prompt Engineering

Classifier Prompt Refinement

The initial classifier prompt caused a misclassification: sudo ... COMMAND=/bin/bash was labelled benign because the 3.2B model treated it as routine admin activity. The prompt was refined to include explicit red-flag rules.

Test 2 (Privilege Escalation) failed on the first run. The fix demonstrates that real agentic systems require iterative prompt engineering — the LLM's reasoning must be guided, not assumed.
❌ Before (v1 — caused failure)
You are a cybersecurity analyst. Classify this log line. Reply with EXACTLY one word: benign, suspicious, or critical. Log: {raw_log} Classification:
✅ After (v2 — all tests pass)
Rules: - benign: routine events (cron, systemd) - suspicious: unusual activity (sudo to shell, brute force) - critical: confirmed attack (uid=0 shell, EXECVE /bin/sh) Red flags — ALWAYS suspicious/critical: - sudo launching /bin/bash or /bin/sh - repeated failed passwords - uid=0 shell execution - EXECVE with /bin/sh or /bin/bash

07 — Reflection

Key Observations

Autonomous Decision-Making

The agent's routing behavior is fully driven by LLM output — there are no hardcoded keyword matches. The same log processed by a weaker prompt yielded a different route, demonstrating that prompt engineering is a first-class concern in agentic system design.

Prompt Engineering as Debugging

Test 2 failing on the first run was a meaningful result, not just an error. It showed that small models (3.2B) need explicit categorical rules and worked examples — implicit reasoning about "unusual" behavior is insufficient. The fix added a red-flag list that guided the model without removing its generative capability.

LangGraph vs. Simple Pipelines

A standard Python function pipeline could achieve the same output for a fixed input. LangGraph adds value through its StateGraph abstraction: the state is explicitly typed, each node is independently testable, and conditional edges make routing logic inspectable and extensible without rewriting the execution engine.

Limitations

The current agent processes one log at a time and does not retain memory across invocations. A production extension would include batch processing, confidence scoring, and a feedback loop where low-confidence classifications are re-evaluated with a stricter prompt before routing.
