Classical Mastery:

ORB-SLAM

Mur-Artal, Montiel, Tardós — IEEE Trans. Robotics, 2015  ·  ~25 min
The canonical realization of the classical pipeline  ·  monocular, real-time, lifelong

The Central Design Choice

  • One feature type for all tasks: tracking, mapping, relocalization, loop closing
  • Prior systems used different features per task — expensive, inconsistent
  • Choice: ORB — Oriented FAST + Rotated BRIEF
  • 256-bit binary descriptor, invariant to rotation & scale
  • Extraction: ~11 ms/frame — no GPU needed
  • SIFT ~300 ms, SURF ~300 ms, A-KAZE ~100 ms — all too slow
Unified features → efficient, simple, reliable. Everything else follows from this.

Three Parallel Threads

① Tracking

  • Every frame — estimates camera pose
  • Motion-only BA
  • Decides keyframe insertion

② Local Mapping

  • Processes new keyframes
  • Triangulates new map points
  • Local bundle adjustment
  • Culls redundant keyframes

③ Loop Closing

  • DBoW2 place recognition
  • Similarity transform (7-DoF)
  • Essential Graph optimization

What Does "Map" Mean in ORB-SLAM?

Map Points
  • 3D position Xw in world coordinates
  • Viewing direction n — mean of all observing rays
  • Best ORB descriptor D — min Hamming distance
  • Scale bounds dmin, dmax — ORB scale invariance range
Keyframes
  • Camera pose Tiw ∈ SE(3)
  • All extracted ORB features (matched or not)
  • Camera intrinsics — focal length, principal point
Map = sparse 3D point cloud + a graph of keyframes. No dense geometry. No color.

Covisibility Graph & Essential Graph

Covisibility Graph
  • Undirected weighted graph over keyframes
  • Edge exists if two KFs share ≥15 map points
  • Edge weight = number of common observations
  • Enables local-map retrieval — focus tracking without scanning full map
Essential Graph
  • Sparser subgraph — retains all nodes, fewer edges
  • Includes: spanning tree + edges with θ ≥ 100 + loop closure edges
  • Used for pose graph optimization after loop closure
  • ~5× keyframe count edges — tight, fast to optimize

  • ORB extraction: 8 scale levels, scale factor 1.2 — homogeneous grid distribution
  • Initial pose: constant velocity model → guided search in previous frame
  • Relocalization: if tracking lost → DBoW2 query → RANSAC PnP
  • Track local map: project map points from covisibility neighbors → refine pose
  • New keyframe condition: visual change >10%, ≥50 tracked points, >20 frames since reloc
Tracking runs at 25–30 Hz. Most demanding step: tracking the local map (~15 ms median)

Thread ②: Local Mapping

  • Keyframe insertion: update covisibility graph, compute BoW representation
  • Map point culling: must be visible in >25% of predicted frames, seen in ≥3 KFs after first 3
  • New point creation: triangulate unmatched ORBs in covisibility neighbors — check parallax, reprojection error, depth
  • Local bundle adjustment: optimize current KF + covisible KFs + their map points jointly
  • Keyframe culling: delete if 90% of its map points visible in ≥3 other KFs at same/finer scale
"Generous in spawning, restrictive in culling" — map stays compact and accurate

Thread ③: Loop Closing

  • Candidate detection: DBoW2 query against DB; require 3 consecutive consistent candidates
  • Similarity transformation: find Sim(3) — 3 translation, 3 rotation, 1 scale — corrects monocular scale drift
  • RANSAC over 3D–3D ORB correspondences (Horn's method)
  • Loop fusion: align both sides of loop, fuse duplicate map points
  • Essential Graph optimization: distribute loop error along sparse subgraph
Why Sim(3) not SE(3)? Monocular SLAM has scale ambiguity — scale can drift over time. Sim(3) corrects it.

Relocalization & Lifelong Map Reuse

Relocalization
  • Triggered when tracking is lost (occlusion, fast motion)
  • Frame → BoW vector → query recognition DB → top candidates
  • RANSAC PnP per candidate → accept if enough inliers
  • High invariance to viewpoint — ORB is rotation & scale invariant
Map Reuse
  • Map persists across sessions — load and localize in existing map
  • Keyframe culling keeps map bounded even in repeated visits
  • Map grows only when new visual content appears
fr2_xyz: ORB-SLAM relocalizes 78% of frames vs PTAM's 35%

Lifelong Operation

  • PTAM: keyframes grow unboundedly over time
  • ORB-SLAM: map size saturates once scene is covered
  • Dynamic changes → new KFs for novel content only

Key numbers

  • Tracking: ~30 ms/frame total (25–30 Hz)
  • Local BA: ~300 ms median (separate thread)
  • Loop detection: sublinear in map size — DBoW2 inverted index
  • Essential Graph opt: <1 s vs 14+ s for full BA

Limitations & What Comes Next

ORB-SLAM limitations
  • Sparse map only — no dense geometry, limited for manipulation or navigation
  • Textureless scenes — corners needed; fails on blank walls, asphalt
  • Scale ambiguity — monocular; scale recovered up to a constant
  • No appearance — cannot render the environment, no photo-realism
  • Map representation is a hard-coded point cloud — not differentiable
The pipeline is sound. What if we replaced the map representation with something richer?

Section 1 — Key Takeaways

  • One feature (ORB) for all tasks — tracking, mapping, relocalization, loop closing — is the key architectural insight
  • Three parallel threads: Tracking (every frame), Local Mapping (per keyframe), Loop Closing (global)
  • Covisibility graph localizes computation — large maps become tractable
  • Essential Graph makes loop correction fast: <1 s vs 72 s for full BA
  • Survival of the fittest: generous spawn, restrictive cull → compact, bounded map
  • Accuracy: typically <1% of path length on KITTI

What "map" means here

  • Sparse 3D point cloud + keyframe graph
  • No density, no color, no differentiability
  • Fast, accurate — but limited expressiveness
ORB-SLAM is the baseline every neural SLAM paper compares against. Know it well.
