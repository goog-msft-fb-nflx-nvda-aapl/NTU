Section 01

Background You Need

This paper sits at the intersection of SLAM and 3D Gaussian Splatting. Here's what each piece means.

🔵

3D Gaussian Splatting (3DGS)

Represent a scene as millions of tiny 3D ellipsoids (Gaussians), each with a position, orientation, size, color, and opacity. Render by "splatting" them onto the image plane — extremely fast via GPU rasterization, not ray marching.

📷

Monocular SLAM

The hardest SLAM setup: only a single RGB camera, no depth sensor. Scale ambiguity is a fundamental challenge — the system must infer depth purely from motion and appearance across frames.

🗺️

Dense vs Sparse SLAM

Sparse SLAM tracks a few hundred keypoints. Dense SLAM reconstructs the entire scene surface. Gaussian SLAM is dense — every Gaussian contributes to both the 3D map and the rendered image used for tracking.

⚡

Direct Optimisation

Instead of extracting keypoints and matching descriptors, direct methods minimise photometric error directly against the 3D representation. More robust to texture-less surfaces, but more computationally demanding.

Key upgrade over neural SLAM: Where iMAP and NICE-SLAM use implicit neural networks (slow to query, needs ray marching), Gaussian SLAM uses explicit 3D Gaussians — directly rasterizable on a GPU at interactive rates.

Section 02 · Foundation

3D Gaussian Splatting — A Quick Primer

Before understanding Gaussian SLAM, you need to understand 3DGS itself. It's a radically different scene representation from NeRF.

What is a Gaussian?

Each 3D Gaussian is parameterised by:

  • μ — center position (x, y, z)
  • Σ — 3D covariance (shape & orientation)
  • α — opacity
  • c — color (via spherical harmonics)
G(x) = exp(−½ (x−μ)ᵀ Σ⁻¹ (x−μ))

Rendering is rasterization: project Gaussians to 2D, sort by depth, alpha-composite. No ray marching. No neural network at render time.

Section 03 · CVPR 2024

Gaussian Splatting SLAM — The Method

Three core innovations make 3DGS work in a live SLAM pipeline.

Innovation 1: Tracking via Direct Optimisation

Original 3DGS required accurate poses from offline SfM as input. That's useless for SLAM.

Gaussian SLAM reformulates tracking as: given the current Gaussian map, find the camera pose T that minimises photometric + depth error between the rendered image and the observed frame.

T* = argmin ‖I_render(T, G) − I_obs‖²

This gives a wide basin of convergence — robust even when the camera moves fast or the scene has few texture features.

Innovation 2: Geometric Verification & Regularisation

Incremental reconstruction creates ambiguities: the same surface may be covered by multiple overlapping Gaussians, or a Gaussian may grow too large and "bleed" into empty space.

Geometric verification — checks that each Gaussian is consistent with observed depth before committing it to the map.

Regularisation — penalises Gaussians that are too large, too elongated, or placed in empty space. Keeps the map tight and accurate.

Innovation 3: Monocular Depth Initialisation

Without a depth sensor, new Gaussians must be initialised from monocular depth estimates (e.g., from a monocular depth network) and then refined via photometric optimisation. The system handles scale ambiguity by anchoring the Gaussian scale to the estimated scene geometry — not assuming any fixed scale.

Result: The same system works with a plain webcam (monocular) or an RGB-D sensor — just swap the depth source. No code changes needed.

What Makes It Special

One representation, everything
The same set of Gaussians is used for tracking, mapping, AND novel view synthesis — no separate modules, no format conversion.

Tiny & transparent objects
Explicit Gaussians can represent very small objects and partially-transparent surfaces — something implicit methods consistently fail at.

Section 05

Check Your Understanding

Four questions on Gaussian SLAM's key ideas.

1. Why does original 3DGS require poses from an offline SfM system, and how does Gaussian SLAM solve this?
✓ Correct! The original 3DGS paper took camera poses as a given input. Gaussian SLAM's key contribution is formulating tracking as direct optimisation against the rendered Gaussian map — making the system fully self-contained without any offline preprocessing.
2. What is "geometric verification" and why is it needed in incremental Gaussian mapping?
✓ Correct! When adding Gaussians frame-by-frame, noisy depth estimates or moving objects can introduce inconsistent Gaussians. Geometric verification rejects Gaussians that don't match the observed geometry, keeping the map clean and accurate.
3. How does Gaussian SLAM handle the scale ambiguity problem in monocular mode?
✓ Correct! Scale ambiguity is fundamental in monocular vision — the same image could have come from a scene twice as far away with objects twice as large. Gaussian SLAM bootstraps scale from a monocular depth network and continuously refines it through photometric optimisation across frames.
4. What is the main advantage of Gaussian Splatting's rasterization renderer over NeRF's ray marching?
✓ Correct! NeRF ray marching evaluates the MLP at hundreds of sample points per pixel — that's millions of forward passes per frame. Gaussian splatting projects ellipsoids to 2D and alpha-composites them, which maps perfectly to GPU rasterization pipelines and achieves 100+ fps on modern hardware.
