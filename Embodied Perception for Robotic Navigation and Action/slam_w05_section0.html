The Embodied Agent's Dilemma

  • To act, an agent needs to know where it is
  • To know where it is, it needs a map
  • To build a map, it must already know where it is
This circular dependency is the SLAM problem
  • Only sensor data available: camera, IMU, LiDAR
  • No GPS indoors, no prior map guaranteed
Let me start with a very simple scenario. Imagine a robot — or a person — dropped into an unfamiliar building. No GPS. No map handed to it. Just a camera. The question is: what does the agent need in order to act? To navigate, to pick something up, to avoid an obstacle? It needs to know where it is. But here is the problem. To know where it is, it needs a map of the environment — some model of the space around it. And to build that map, it needs to know where it was when it saw each part of the world, which means it needs its own position. Look at the diagram on the right. The three nodes — ACT, POSE, and MAP — form a triangle. POSE feeds ACT. MAP feeds ACT. But POSE and MAP depend on each other. The dashed arrows in the middle represent that circular dependency. This is not a software bug. It is a fundamental geometric constraint. The only input the agent has is raw sensor data — pixels from a camera, maybe inertial measurements. Everything else must be inferred simultaneously. This is the SLAM problem. Simultaneous Localization And Mapping. Solve both at once, from scratch.

Two Sub-problems, One Joint Task

Localization

Where am I?

  • Estimate pose xt given map M
  • Kalman filter, particle filter, PnP solve
  • Easy if map is known
Mapping

What does the world look like?

  • Build M from observations z1:t
  • Occupancy grid, point cloud, mesh, implicit
  • Easy if pose is known
SLAM solves both simultaneously from scratch — no prior pose, no prior map
Let's separate the two sub-problems so we can be precise about what makes each one hard. Localization asks: where am I? Formally, given a map M that I already have, estimate my current pose — my position and orientation in 3D space. If the map is given to you, this is a well-studied problem. You can use a Kalman filter, a particle filter, or a direct geometric solver like PnP. Hard in practice, but tractable in principle. Mapping asks: what does the world look like? Given my poses — which I already know — build a model M from what I have observed. Again, if the poses are given, this is also tractable. You triangulate landmarks, you fuse depth measurements, you build an occupancy grid or a point cloud. The catch is the italicized condition in each case. Localization is easy *if* the map is known. Mapping is easy *if* the pose is known. In SLAM, neither is known. You have to solve them together, iteratively refining your estimate of both. Look at the diagram on the right. The blue line is the robot's estimated trajectory. The orange dots are landmarks the robot has observed. The dashed lines connect each robot position to the landmarks it saw at that moment. Everything in that picture — every dot, every line — is an unknown that SLAM must estimate jointly from the raw pixel stream. This joint estimation is what gives SLAM its name, and its difficulty.

Factor Graphs: The Probabilistic View

  • SLAM = MAP inference over poses & landmarks
  • Factor graph separates variables from constraints
  • Variable nodes — poses xt, landmark positions lk
  • Factor nodes — odometry, observations, priors
  • Optimization: minimize sum of squared errors
Sparse structure → efficient solvers (g2o, GTSAM, Ceres)
Now let's give this problem a mathematical shape. The standard formalism is the factor graph. A factor graph has two kinds of nodes. Variable nodes represent the unknowns we want to estimate — the robot's pose at each timestep, x-one, x-two, x-three, and the positions of landmarks in the world, l-one. These are the circles in the diagram. Factor nodes represent constraints — pieces of information that relate the variables to each other. There are two types here. The green squares are odometry factors: they encode what the robot's wheel encoders or IMU said about how it moved between two consecutive poses. The grey squares are observation factors: they encode what the camera saw — a landmark appeared at a certain pixel location, which constrains the relationship between the robot's pose and the landmark's position. SLAM is then formulated as maximum a posteriori inference over this graph — find the assignment of all variables that best satisfies all the constraints simultaneously. In practice this becomes a nonlinear least-squares problem: minimize the sum of squared residuals across all factors. The key insight is that this graph is sparse. Each pose is only connected to its neighbors and to the few landmarks it observed. That sparsity is what makes the problem tractable — solvers like g2o, GTSAM, and Ceres exploit it heavily, and they can optimize graphs with thousands of variables in real time. When we look at ORB-SLAM in the next section, you will see exactly this structure — a pose graph and a landmark set — being built and optimized continuously as the camera moves.

Loop Closure: Why It Matters

  • Odometry drift accumulates over time — errors compound
  • Loop closure: agent recognizes a previously visited place
  • Adds a long-range constraint → corrects accumulated drift
  • Without it: map folds, trajectories diverge
  • Recognition via appearance (bag-of-words, DBoW) or geometry
Loop closure is what makes SLAM globally consistent
There is one phenomenon that separates a SLAM system that works in small spaces from one that works in a building or a city. It is called loop closure. Here is the problem it solves. Every sensor measurement has noise. When you integrate noisy odometry over time — every small step, every small turn — the errors accumulate. After a hundred meters, your estimated position might be off by several meters. After a kilometer, by tens of meters. The trajectory drifts. This is unavoidable with odometry alone. Now look at the top diagram. The robot starts at the green dot, travels the dashed orange path, and ends up at the orange dot — which is nowhere near where it started, even though it has returned to the same physical location. The map it built along the way is geometrically inconsistent. Loop closure fixes this. At some point, the robot recognizes that it is looking at a place it has seen before. Maybe the same corner, the same doorway, the same set of feature points. This recognition adds a hard constraint to the factor graph: the current pose and the earlier pose must be close together, because they are both looking at the same place. When you add that constraint and re-optimize the graph, the entire trajectory snaps into a consistent shape — shown in the lower diagram in blue. The loop constraint, shown as the dashed green line, pulls the end of the trajectory back toward the start. This is what makes SLAM globally consistent. Without loop closure, you have a locally consistent map that slowly falls apart over distance. With loop closure, you have a map that self-corrects every time the robot revisits known territory. The hard part, which ORB-SLAM handles very cleverly, is the recognition step: how do you reliably identify that you have seen this place before, without false positives?

Next section: ORB-SLAM is the canonical realization of this pipeline — we'll trace every stage through its code & figures
Let me now put the pieces together into the classical SLAM pipeline — the architecture that dominated the field for about a decade, and that ORB-SLAM brought to its highest form. The pipeline flows left to right. It starts with sensor input — typically a monocular camera, a stereo pair, or an RGB-D sensor. The first thing you do with each new frame is extract features. In ORB-SLAM, those are ORB features — Oriented FAST and Rotated BRIEF — which are rotation-invariant, fast to compute, and fast to match. Next is data association: you take the features from the current frame and match them to features you have seen before — either in the previous frame, or in the existing map. This step is where most failures happen in classical systems; a bad match leads to a bad pose estimate. With good matches established, you run pose estimation — typically a PnP solve, which finds the camera pose that best explains where the matched 3D map points project in the current image. Once you have a pose, you update the map — add any new landmarks that appeared in this frame, refine the positions of existing ones. And then, running in a separate thread at lower frequency, loop closure continuously monitors for place revisits. When one is detected, it adds a constraint to the graph and triggers a global bundle adjustment to correct all accumulated drift. This is the skeleton. Every SLAM system you will see today — classical or neural — is a variation on this skeleton. The question is always: what represents the map? What does feature extraction look like? How is the optimization structured? In the next section, we will go deep on ORB-SLAM, which is the gold standard answer to all of those questions in the classical regime.

Section 0 — Key Takeaways

  • SLAM = simultaneous localization and mapping — circular dependency solved jointly
  • Factor graphs formalize SLAM as a sparse nonlinear least-squares problem
  • Loop closure is essential for global consistency — without it, drift is unbounded
  • Classical pipeline: features → association → pose → map → loop close
  • Modern variants replace each block with neural representations

"The map is not the territory — but for a robot, building the map is navigating the territory"
This slide stays up during questions, so I will keep this brief as a verbal summary. SLAM is the problem of building a map and estimating your own position at the same time, from sensor data alone. The circular dependency — you need a map to localize, you need a pose to map — is resolved by solving both jointly as a probabilistic inference problem. Factor graphs give us the right language: variable nodes for poses and landmarks, factor nodes for motion and observation constraints, and a sparse optimization that minimizes the total error across all constraints. Loop closure is the mechanism that prevents drift from destroying the map over long trajectories. It is not optional — without it, any real-world deployment will fail. The classical pipeline packages all of this into a sequence of well-defined stages: extract features, associate them, estimate the pose, update the map, detect loops. ORB-SLAM is the canonical realization of that pipeline. And the key insight for today's lecture is that the map representation — what "M" actually is — is a design choice. Classical systems use sparse point clouds. Neural systems use MLPs or feature grids. Gaussian systems use 3D Gaussians. The pipeline stays roughly the same. The representation changes everything about how you store, query, and render the map. Any questions before we move on to ORB-SLAM?
